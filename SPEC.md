# 招待リンク型オンライン古典ゲーム集サイト 仕様書

## 1. 目的

本サービスは、招待リンクでオンライン対戦できる古典的なゲームをまとめた Web サイトである。
ユーザーはブラウザからアクセスし、アカウント登録なしでもルームを作成・参加でき、短時間で対戦を開始できることを重視する。

初期リリースでは、実装コストが低く、ルールが分かりやすく、同時接続対戦に向いたゲームを提供する。
本サービスではログイン機能は提供せず、表示名とセッション識別子のみでプレイできる設計とする。

## 2. サービス概要

- サービス名: 未定
- 提供形態: PC / スマートフォンブラウザ対応 Web アプリ
- コンセプト: 懐かしい定番ゲームを、招待リンクですぐ対戦できる場所
- 主な価値:
  - インストール不要
  - 会員登録なしでも開始可能
  - ルーム作成から対戦開始までが速い
  - 複数の古典ゲームを1サイトに集約

## 3. 対象ユーザー

- 友人同士で軽く遊びたいユーザー
- 通話しながら対戦したいユーザー
- インストールや面倒な設定を避けたいユーザー
- シンプルな対戦ゲームを探しているユーザー

## 4. 対象プラットフォーム

- 対応ブラウザ:
  - Chrome 最新版
  - Safari 最新版
  - Edge 最新版
- 対応デバイス:
  - PC
  - スマートフォン
- 優先対応:

  - スマートフォンはプレイ可能レベルを担保

- サーバー同期が比較的シンプル
- 1プレイが 1 分から 10 分程度で終わる
- 4 人前提の古典ゲームは bot 補充で成立させられる

## 6. 基本ユースケース

### 6.1 ゲストユーザーが友人と対戦する

1. トップページでゲームを選ぶ
2. ルームを作成する
3. 招待リンクを共有する
4. 相手がリンクから参加する
5. 必要人数が揃ったらゲーム開始
6. 勝敗表示後、同じルームで再戦できる

### 6.2 ゲストユーザーが bot 補充ありのトランプゲームで対戦する

1. ゲームを選ぶ
2. ルームを作成する
3. 必要人数に満たない席へ bot を補充する設定を選ぶ
4. 招待リンクで相手を呼ぶ
5. 人間プレイヤーと bot を含めてゲーム開始する

### 6.3 再接続したユーザーが対戦へ復帰する

1. 通信切断またはリロードが発生する
2. 同一セッション識別子でルームへ再接続する
3. 最新のゲーム状態を復元して対戦へ復帰する

## 7. 機能要件

### 7.1 共通機能

- ゲーム一覧表示
- 各ゲームのルール説明表示
- ルーム作成
- 招待リンク発行
- ルーム参加
- プレイヤー名入力
- 対戦開始
- 再戦
- 離脱時の勝敗処理
- エラー表示

### 7.2 プレイヤー識別機能

- アカウント登録なしでゲスト利用可能
- 任意の表示名設定
- セッション単位のプレイヤー識別子を発行
- ブラウザ再読み込み時に同一セッションとして再接続可能
- 永続的なユーザープロフィールは持たない

### 7.3 マッチング機能

- 友達招待用のプライベートルーム
- ルーム状態管理
  - 募集中
  - 対戦中
  - 対戦終了
  - 切断
- 満室時の参加拒否
- ゲームごとの想定人数を超えるプレイヤーは参加できないこと
- bot 席を持つゲームでは bot の割り当てができること
- 対戦開始後の新規参加は拒否し、既存セッションのみ再接続できること

### 7.4 リアルタイム通信機能

- プレイヤー入室通知
- 手番更新
- 盤面同期
- 勝敗通知
- 再戦意思通知
- 切断通知
- 再接続時の状態復元

### 7.5 ゲーム機能

各ゲームは以下を満たす。

- ゲーム開始時に手番順または初期行動順を決定できる
- 現在手番を表示する
- 非合法手を拒否する
- 勝敗または引き分けを判定する
- 終了後に結果を表示する
- タイムアウトルールを設定可能にする余地を残す
- カードゲームでは配札、場札、手札、捨て札をサーバー側で厳密に管理する
- 非公開情報を持つゲームでは、プレイヤーごとに異なる表示状態を返す
- bot 対応ゲームでは bot の思考ターンをサーバー側で実行する

### 7.6 履歴機能

- ルーム内の対戦結果はそのセッション中のみ参照可能
- 永続的な個人戦績保存は行わない
- 集計ランキングやレーティングは提供しない

### 7.7 管理機能

- ゲームの公開/非公開切替
- 通報された表示名の確認
- 問題ルームの強制終了
- お知らせ管理

## 8. 非機能要件

### 8.1 パフォーマンス

- ルーム参加から対戦開始まで 3 秒以内を目標
- 通信イベント反映は通常 300ms 以内を目標
- トップページ初回表示は 2 秒以内を目標

### 8.2 可用性

- 短時間の切断から再接続できる
- 片方のプレイヤー離脱時に状態不整合が起きない

### 8.3 セキュリティ

- 招待 URL は推測されにくいランダム値を使う
- 表示名は文字数制限と禁止ワード制御を行う
- API にレート制限を入れる
- WebSocket 接続時にルーム権限を検証する
- 手札などの秘匿情報は該当プレイヤーにのみ送信する

### 8.4 保守性

- ゲームロジックはゲームごとに分離する
- 共通のルーム管理層と個別ゲームルール層を分ける
- 新しいゲームを追加しやすい設計にする
- Cloudflare Workers 上で動作する制約を前提に、Node.js 固有 API に依存しない

## 10. ルーム仕様

- 基本は人間 2 人から開始できる設計とする
- ゲームごとに `seatCount` を持ち、対応人数の範囲内でルーム人数を選べる
- bot 対応ゲームは設定に応じて空席を bot で補充できる
- スペードや七並べなど複数人数前提ゲームは bot を補充して必要席数を満たす
- ルーム作成時に以下を保持する
  - ルーム ID
  - ゲーム種別
  - 作成者
  - 参加者
  - bot 参加席情報
  - ルーム状態
  - 現在のゲーム状態
  - 作成日時
- 一定時間参加者が揃わないルームは自動破棄
- 対戦終了後、同一メンバーで再戦可能
- ルームオーナー不在でも対戦継続可

## 11. ゲーム別仕様概要

### 11.1 オセロ

- 8x8 盤面
- 初期配置あり
- 石を置けない場合はパス
- 両者置けない時点で終了
- 石数が多い側が勝利

### 11.2 五目並べ

- 15x15 盤面を基本とする
- 先に 5 つ連続で並べた側が勝利
- MVP では禁じ手なしの簡易ルール

### 11.3 四目並べ

- 7 列 x 6 行
- 下に落ちる形式
- 先に 4 つ揃えた側が勝利

### 11.4 じゃんけん

- 2 人から 6 人まで対応する
- 全員が同時に手を選ぶ
- 出た手が 2 種類のときのみ勝敗判定を行う
- 全員同手、または 3 種類すべて出た場合はあいことして次ラウンドへ進む
- 勝者が複数人になる場合がある
- 連戦しやすいテンポを重視

### 11.5 ババ抜き

- 標準 52 枚からジョーカー 1 枚を含める
- 2 人から 4 人まで対応する
- ルーム設定に応じて空席へ bot を追加できる
- 同じ数字のペアは自動で場から除外する
- 手番プレイヤーは左隣の有効な手札から 1 枚引く
- bot の手番は Worker 側で自動進行する
- 最後にジョーカーを持っていたプレイヤーが敗北

### 11.6 七並べ

- 4 スートの 7 を初期配置として開始する
- 人間 2 人を基本とし、不足席は bot で補充する
- パス回数上限の有無はルーム設定で切替可能にする余地を残す
- 手札を先に出し切った順で順位を確定する

### 11.7 スペード

- 4 人制の古典ルールを基本とし、人間 2 人 + bot 2 体を標準構成とする
- 2 対 2 の固定ペア戦とする
- 各ラウンド開始時にビッドを宣言する
- トリック獲得数とビッド達成状況から得点を計算する
- bot は合法手判定と基本戦略に基づいて自動で行動する

## 12. 技術仕様案

### 12.1 フロントエンド

- React
- TypeScript
- 状態管理は React 標準を優先し、必要に応じて外部ライブラリを採用
- WebSocket クライアントでリアルタイム同期
- Cloudflare Workers から配信する静的アセット構成を前提とする

### 12.2 バックエンド

- Cloudflare Workers
- Durable Objects をゲームルーム単位の単一調停役として利用する
- Durable Objects 上で WebSocket 接続を受け、ルーム状態を管理する
- REST API または RPC で補助機能を提供
- ゲームルーム管理とゲームロジック検証を担当
- bot の意思決定も Worker 側で実行する

### 12.3 データストア

#### MVP

- Durable Objects のストレージをルーム状態の保持に利用
- KV をルール文面や軽量な設定値保持に利用する余地を残す

#### 将来拡張

- D1 を使って公開用メタデータや運営設定を永続化する余地を残す

### 12.4 インフラ

- Cloudflare Workers にアプリケーション本体を配置
- Durable Objects でルーム単位の状態とリアルタイム同期を処理
- Cloudflare CDN 配信
- 監視、ログ収集の導入

### 12.5 Workers 設計方針

- 1 ルーム = 1 Durable Object を基本とする
- ルーム内の状態更新は Durable Object 内で逐次処理する
- WebSocket Hibernation を活用し、待機中ルームのコストを抑える
- 長期永続化が不要な対戦データはルーム終了後に破棄する
- Workers ランタイム互換のライブラリのみ採用する

## 15. バリデーション要件

- 表示名は 2 文字以上 20 文字以下
- 不正なゲーム操作はサーバー側で拒否
- 手番でないプレイヤーの操作は無効
- 終了済みゲームへの追加操作は無効
- 参加していないルームへの操作は拒否
- bot 席に対するクライアントからの操作は拒否
- 非公開カード情報を他プレイヤーに送信しない

## 16. エッジケース

- プレイヤーが手番中に切断した
- 相手不在のままルームが放置された
- 両者が同時に再戦を押した
- 同時送信により競合する操作が届いた
- ブラウザ再読み込み後に再参加した
- 同じ URL を複数人が開いた
- bot を含むゲームで人間プレイヤーだけが離脱した
- 手札の非公開情報が再接続後も正しく復元される

## 20. 受け入れ条件

- ユーザーがトップページから任意ゲームを選び、5 分以内に対戦を開始できる
- 参加中プレイヤーのブラウザ間でゲーム状態が同期される
- 不正な手がサーバーで拒否される
- 勝敗が正しく表示される
- 対戦終了後に再戦できる
- 片方が切断した場合、相手側に状態が通知される
- じゃんけんでは 2 人から 6 人で対戦が成立する
- ババ抜きでは 2 人から 4 人で対戦が成立し、bot を含む進行も継続できる
- 秘匿情報を持つカードゲームで他プレイヤーの手札が露出しない
