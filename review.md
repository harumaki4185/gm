### 🟠 Major

#### M-V15-1: ロン候補の探索がすべての座席を対象にしていない（ダブロン未対応）

**ファイル:** `worker/games/mahjong.ts:771-786`

```typescript
const ronSeat = turnOrder.find(    // ← find は最初の1人しか返さない
  (seat) =>
    evaluateMahjongWin(room, state, seat, "ron", discardTile, discardSeat) !== null
);
if (typeof ronSeat === "number") {
  return {
    seat: ronSeat,
    ...
    canRon: true,
    chiOptions: [],
    ponOption: null,
    kanOption: null
  };
}
```

`Array.find` のため、複数人がロン可能でも先頭1人にしか `pendingCall` が届きません。MAHJONG_SPEC.md §6.4 / §2.3 でダブロン（複数和了）が「あり」と定義されているため、仕様との乖離があります。

**影響:** ダブロン局面で2人目にロン機会が与えられないままゲームが続く（または1人目がロンすると終了）。

**対応案:** `turnOrder.filter(...)` で全和了者リストを収集し、`pendingCall` を複数人対応に拡張するか、最優先ロン者一人に絞る設計であれば仕様書（MAHJONG_SPEC.md）の記述を修正する。

### 🟡 Minor

#### m-V15-2: `shouldBotClaimTriplet` の判断ロジックに論理的に不自然な部分がある

**ファイル:** `worker/games/mahjong.ts:402-425`

カン（大明槓）の判断で `evaluateMahjongWin(... "tsumo" ...)` を呼んでいますが、これはカン前の現在の手牌でツモ和了できるかを確認するもので、カンすべきかどうかの判断には結びつきません（カン後の嶺上和了は別の翻数になる）。bot の判断精度の問題であり、ゲームの進行には支障ありません。

`src/shared/games.ts:107` の `category: "table"` が型定義に存在しない点、引き続き対応をお勧めします。
